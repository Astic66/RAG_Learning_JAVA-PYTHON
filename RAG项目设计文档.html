1. 项目概述

本项目是一个本地部署的 RAG（Retrieval-Augmented Generation）企业内部知识问答系统。它允许用户上传企业文档（如规章制度、产品手册、培训材料），系统自动对文档进行切块和向量化，并存储在本地 JSON 向量库中。用户可以通过自然语言提问，系统先从知识库中检索最相关的文本片段，再调用本地大语言模型基于检索结果生成准确、可溯源的回答。

核心能力

文档上传、文本切块、Embedding 向量化、余弦相似度检索、LLM 增强生成、来源展示、对话历史。

典型场景

企业制度查询、产品 FAQ、内部培训问答、课程笔记检索、个人知识库助手。

4. 目录结构设计

项目采用前后端分离、服务层内聚的组织方式。两套后端实现共享前端和存储目录。

6. 前端设计

6.1 页面布局

采用经典的三栏式布局：

  • 顶部栏：项目名称 + 后端连接状态
  • 左侧知识库面板：文件上传、已上传文档列表、删除索引
  • 右侧问答面板：对话消息气泡区、输入框、提问按钮

6.2 交互流程

用户操作前端行为后端接口
选择文件并点击上传通过 FormData 发送文件，显示加载状态POST /api/documents
输入问题并点击提问渲染用户气泡，显示"思考中"，请求接口POST /api/chat
收到回答渲染 AI 气泡，展开/折叠来源卡片-
点击删除索引发送 DELETE 请求，刷新文档列表DELETE /api/documents/{id}

6.3 状态管理

前端使用原生 JavaScript 管理状态：

  • messages 数组保存完整对话历史
  • 对话历史通过 localStorage 持久化，刷新页面不丢失
  • 每次渲染时根据 role 字段分别渲染用户/AI/加载三种气泡

7.3 关键算法说明

文本分块

采用滑动窗口策略，窗口大小 800 字符，步长 680 字符（800 - 120 重叠），保留 120 字符重叠。目的是防止句子在块边界被切断，提升检索召回率。

余弦相似度

cos_sim(A, B) = (A · B) / (||A|| × ||B||) 其中 A · B 表示向量点积，||A|| 表示向量 L2 范数。 结果越接近 1，表示两段文本语义越相似。

检索策略

暴力线性扫描全部向量并计算相似度，排序后取 Top-K（默认 K=4）。数据量小时足够；数据量大时可替换为专业向量数据库（如 FAISS、Milvus、Chroma）。

8. API 接口设计

RESTful 风格接口，所有响应均为 JSON。当前版本为 v1 内嵌设计，未加版本前缀。

方法路径请求体响应说明
GET/api/health-status, chat_model, embedding_model健康检查
GET/api/documents-文档列表（含块数）列出知识库文档
POST/api/documentsmultipart/form-data: filedocument_id, file_name, chunk_count上传并建立索引
DELETE/api/documents/{id}-deleted_chunks删除文档索引
POST/api/chat{question, top_k}{answer, sources[]}提问并生成答案
GET/ / /assets/*-HTML / CSS / JS前端静态资源
接口设计说明：接口数量精简，职责单一。上传接口同时完成文件保存、分块、Embedding、入库，简化前端调用；问答接口同时完成检索和生成，一次性返回答案与来源。

9. 数据存储设计

9.1 向量库结构

向量库存储在 backend/storage/vector_store.json，为 JSON 数组。每个元素代表一个文本块：

{ "id": "uuid-单个块唯一ID", "document_id": "uuid-所属文档ID", "document_name": "示例知识库.txt", "chunk_index": 0, "content": "员工出差用餐报销上限为每餐500元...", "embedding": [0.123, -0.456, 0.789, ...] }

9.2 原始文档存储

上传的原始文件保存在 backend/storage/documents/，仅用于备份和下载，不参与检索。检索完全依赖向量库中的向量和文本。

9.3 存储设计取舍

方案优点缺点本项目选择
JSON 文件零依赖、直观、便于学习数据量大时性能差✅ 当前使用
FAISS / Chroma检索快、支持海量向量增加依赖和复杂度⭕ 推荐扩展
MySQL / PostgreSQL事务性强不适合高维向量检索❌ 不适合

10. 部署与使用

10.1 环境要求

  • Python 3.x
  • Ollama 已安装并运行
  • 已下载 qwen2.5:7b 和 nomic-embed-text 模型

10.2 启动步骤

# 1. 确认模型 ollama list # 2. 启动后端（方式一：双击脚本） 启动后端.bat # 3. 启动后端（方式二：命令行） cd backend python server.py # 4. 打开浏览器访问 http://127.0.0.1:8000

10.3 使用流程

  1. 在左侧上传企业文档（txt / md）
  2. 等待系统完成切块和向量化
  3. 在右侧输入问题，按回车或点击提问
  4. 查看 AI 回答，点击来源卡片可展开原始片段

11. 学习价值与扩展方向

11.1 适合学习的点

  • 理解 RAG 完整链路：分块 → Embedding → 检索 → 生成
  • 学习如何手写余弦相似度和 multipart 解析
  • 学习零依赖后端如何服务前端静态资源和 REST API
  • 理解前后端分离项目的基本组织方式

11.2 推荐扩展方向

方向具体做法
多格式支持切换到 FastAPI 版，启用 PDF / DOCX 解析
向量库升级用 Chroma / FAISS 替换 JSON 向量库
知识图谱 RAG提取实体关系，结合图数据库做混合检索
流式输出将 Ollama stream 改为 true，前端逐字显示
会话管理后端保存多轮对话上下文，支持多会话

12. 风险与注意事项

  • 性能上限：JSON 向量库使用暴力扫描，文档量大时检索会变慢。
  • 并发能力：标准库 http.server 仅适合学习和轻量使用，不建议直接生产。
  • 答案可靠性：LLM 仍可能"幻觉"，系统提示词要求基于资料回答，但需人工校验关键信息。
  • 文件编码：当前默认按 UTF-8 解码，非 UTF-8 文件可能乱码。
  • 本地运行：Ollama 需要充足显存/内存，7B 模型建议至少 8GB 显存或 16GB 内存。
